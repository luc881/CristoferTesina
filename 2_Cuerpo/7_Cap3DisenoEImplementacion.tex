\section{Arquitectura general del sistema}

El diagrama de bloques muestra los tres componentes principales del sistema y los canales de comunicación entre ellos: el nodo sensor instalado en la farmacia, el servidor y base de datos alojados en la nube, y la interfaz web accesible desde el navegador. Cada componente corresponde a uno de los objetivos específicos del proyecto y se describe en detalle en las secciones siguientes.

El diagrama de casos de uso muestra quiénes interactúan con el sistema y qué funciones tiene disponibles cada uno. Cada caja con encabezado oscuro representa un actor o una función del sistema; las \textbf{líneas sólidas} indican quién tiene acceso a qué; las \textbf{flechas punteadas «extend»} señalan funciones opcionales que amplían una función base; y la \textbf{flecha con triángulo} entre los actores humanos indica que el Administrador hereda todas las funciones del Cajero y añade las propias.

El \textbf{Cajero} opera el punto de venta: inicia y cierra sesión, registra ventas, consulta el historial y visualiza el sensor. El \textbf{Administrador} hereda todas las funciones del Cajero (flecha con triángulo) y añade las propias: gestión del catálogo, existencias y lotes, compras y usuarios. El \textbf{Sensor ESP32} envía datos de temperatura y humedad automáticamente cada 30 segundos. Las funciones «extend» son opcionales: el escáner puede activarse al registrar una venta; la recuperación de contraseña extiende el inicio de sesión; y la devolución se aplica desde el historial.

\textbf{Rol} y \textbf{Usuario} gestionan los accesos. \textbf{LecturaSensor} almacena cada medición del ESP32. El flujo principal de una venta va de \textbf{Venta} a \textbf{DetalleVenta} y de ahí a \textbf{UsoLoteVenta}, que registra qué lote específico se descontó del inventario — esto implementa el criterio FEFO. \textbf{Producto} y \textbf{LoteProducto} modelan el inventario; cada producto tiene múltiples lotes con distintas fechas de vencimiento. \textbf{Proveedor} y \textbf{Compra} registran las adquisiciones de mercancía.

La interfaz web es el punto de entrada para el personal de la farmacia. Las figuras siguientes muestran el flujo de navegación completo: inicio de sesión, panel principal, historial de ventas, registro de una venta con lector de código de barras y los módulos adicionales del sistema.

\section{Desarrollo de la base de datos en PostgreSQL}

Para cumplir con el primer objetivo específico se diseñó e implementó una base de datos relacional en PostgreSQL que centraliza toda la información del sistema: productos, lotes de inventario, usuarios, ventas y mediciones ambientales. Una base de datos relacional organiza la información en tablas vinculadas entre sí mediante referencias, lo que permite consultar datos relacionados de forma eficiente y mantener la consistencia de la información. El esquema resultante comprende \textbf{22 tablas} agrupadas en cinco módulos funcionales.

\subsection{Selección de PostgreSQL}

  Se eligió PostgreSQL como sistema gestor de base de datos por ser la opción
  recomendada por los ecosistemas de las herramientas utilizadas en este proyecto
  y por su amplia adopción en el desarrollo de aplicaciones
  web profesionales. Frente a alternativas como MySQL o SQLite, ofrece una ventaja
  concreta para este sistema.
  
  Esta es, el tipo de dato \textbf{NUMERIC}, que almacena cantidades monetarias
  con precisión exacta, sin los errores de redondeo que presentan los tipos de punto
  flotante al acumular operaciones. En un sistema que calcula totales de ventas y
  maneja precios con decimales, esta precisión es necesaria para garantizar que los
  montos registrados sean exactos.

\subsection{Organización modular del esquema}

Las 22 tablas se dividieron en cinco módulos según el área del negocio que gestionan, de modo que cada módulo tenga una responsabilidad clara y las dependencias entre módulos sean explícitas:

\begin{enumerate}[nosep, itemsep=3pt]
    \item \textbf{Autenticación y control de acceso:} usuarios, roles y permisos.
    \item \textbf{Catálogo de productos:} información de los medicamentos y sus características.
    \item \textbf{Inventario y compras:} lotes, fechas de caducidad y órdenes de abastecimiento.
    \item \textbf{Registro de ventas:} transacciones, pagos y devoluciones.
    \item \textbf{Mediciones del sensor:} lecturas de temperatura y humedad.
\end{enumerate}

Esta separación hace que las tablas de un módulo puedan crecer o modificarse sin impactar innecesariamente a los demás, y permite localizar con facilidad qué parte del esquema es responsable de cada tipo de dato.

\subsection{Módulo 1: Autenticación y control de acceso}

Este módulo agrupa las cinco tablas relacionadas con el acceso al sistema: usuarios, roles, permisos, la relación entre roles y permisos, y sucursales.

El control de acceso se estructuró por roles: cada usuario pertenece a un rol —por ejemplo, \textit{administrador} o \textit{cajero}— y el rol define qué operaciones puede realizar. Esto simplifica la administración: cambiar los privilegios de un grupo de usuarios requiere modificar únicamente el rol, no cada cuenta individualmente.

Para los registros que no deben eliminarse permanentemente —usuarios, roles y sucursales— se optó por conservar el registro en la base de datos y anotar la fecha de baja en una columna adicional. Esto es necesario porque el historial de ventas referencia al usuario que realizó cada transacción; eliminar físicamente ese registro rompería esa relación.

\subsection{Módulo 2: Catálogo de productos}

Este módulo gestiona la información de los medicamentos a través de seis tablas: el catálogo general de productos, los medicamentos genéricos, las categorías, las marcas, los ingredientes activos y la relación entre productos e ingredientes.

Se separó el medicamento genérico de sus presentaciones comerciales. Por ejemplo, \textit{Paracetamol} existe como un único registro genérico, y cada presentación concreta —\textit{Paracetamol 500~mg tabletas}, \textit{Paracetamol suspensión 250~mg/5~ml}— es un registro distinto que apunta a ese genérico. Esto evita repetir el nombre del principio activo en cada presentación y permite agrupar reportes por nombre genérico sin trabajo adicional.

Las categorías se organizan en una jerarquía de niveles: una categoría puede contener subcategorías, y estas pueden contener otras más específicas (\textit{Medicamentos} → \textit{Analgésicos} → \textit{Antiinflamatorios}). Todo esto se almacena en una sola tabla, sin límite fijo de profundidad.

La relación entre productos e ingredientes activos se registra con la cantidad y unidad de medida de cada componente. Guardar esta información en la base de datos —no solo en la etiqueta física— permite buscar o filtrar medicamentos por principio activo directamente desde la aplicación.

\subsection{Módulo 3: Inventario y compras}

Este módulo controla las existencias físicas y el abastecimiento. Sus tablas registran los lotes de cada producto, los proveedores y las órdenes de compra con el detalle de cada línea.

El inventario se gestiona a nivel de \textbf{lote}, no solo por producto. Cada lote tiene su propio código, fecha de caducidad y cantidad disponible. Esto es necesario en una farmacia porque las regulaciones de buenas prácticas de almacenamiento establecen que los medicamentos deben despacharse en orden de caducidad: primero los lotes que vencen antes. Sin este desglose por lote, el sistema sabría cuántas unidades hay de un producto, pero no podría determinar cuál lote se está consumiendo en cada venta.

Se identificó durante el análisis una situación concreta: algunos medicamentos ingresan a la farmacia sin número de lote asignado por el fabricante. Para no bloquear ese registro, la restricción de unicidad sobre el código de lote se configuró de forma que aplique solo cuando el código existe —si no hay código, se permiten múltiples entradas del mismo producto sin conflicto. Cuando sí hay código, no se permite duplicarlo para el mismo producto, previniendo registros erróneos.

\subsection{Módulo 4: Registro de ventas}

Este módulo registra las transacciones comerciales a través de cinco tablas: la venta en sí, el detalle de productos incluidos, el vínculo con los lotes descontados, los pagos y las devoluciones.

Cada venta se divide en dos niveles: un encabezado con los datos generales de la transacción —empleado, sucursal, descuento y total— y líneas de detalle con cada producto, su cantidad y precio unitario. Esta separación permite que una sola venta contenga múltiples productos sin repetir los datos generales en cada línea.

Adicionalmente, cada línea de detalle queda vinculada al lote específico del que se descontaron las unidades. Esto permite rastrear, dado un lote, en qué ventas fue despachado. En el contexto farmacéutico es relevante ante un retiro de mercado: el sistema puede identificar exactamente qué ventas contienen el lote afectado.

La venta puede encontrarse en uno de tres estados: en progreso, completada o cancelada. El descuento del inventario ocurre únicamente al confirmar la transacción, no antes. Esto evita que una venta que no se concreta descuente existencias.

Los pagos se registran en una tabla separada para soportar el cobro con más de un método: una transacción puede liquidarse parte en efectivo y parte con tarjeta, registrando cada pago de forma independiente.

\section{Desarrollo de la API REST con FastAPI}

Para cumplir con el segundo objetivo específico se diseñó e implementó una API REST en Python con el \textit{framework} FastAPI. La API actúa como intermediario entre los clientes y la base de datos: recibe peticiones, las procesa y devuelve la información en un formato que ambos clientes pueden interpretar. El resultado expone operaciones para gestionar inventario, ventas, usuarios, compras y lecturas del sensor, organizadas en 23 grupos de rutas.

\subsection{Selección de FastAPI}

Se eligió FastAPI sobre alternativas como Flask o Django por dos razones concretas.

La primera es que verifica automáticamente que los datos que llegan en cada petición sean correctos antes de procesarlos. Si algo no cumple lo esperado, rechaza la petición con un mensaje descriptivo sin necesidad de escribir ese código de validación manualmente.

La segunda es que genera automáticamente una página de documentación interactiva donde se puede ver cada operación disponible, sus parámetros y las posibles respuestas. Esto permitió verificar el funcionamiento de la API directamente desde el navegador durante el desarrollo, sin herramientas adicionales.

\subsection{Arquitectura en capas}

La API funciona como una serie de filtros por los que pasa cada petición antes de poder interactuar con la base de datos. Se separó en filtros distintos porque cada uno tiene una responsabilidad diferente: si en algún momento cambia una regla de seguridad o una validación del negocio, solo se modifica ese filtro sin tocar el resto.

El primer filtro actúa como un enrutador: dependiendo de la dirección de la petición, determina a qué operación corresponde y la redirige al área correspondiente, de forma similar a como un router de red decide por qué camino enviar el tráfico. El segundo filtro verifica que el usuario esté identificado y que tenga permiso para realizar esa operación; si no cumple alguna de las dos condiciones, la petición es rechazada sin continuar. El tercero aplica las reglas del negocio, como confirmar que hay existencias suficientes antes de registrar una venta. El último ejecuta la operación sobre la base de datos y devuelve el resultado al cliente.

Para interactuar con la base de datos desde el código Python, cada tabla del esquema descrito en el OE1 se representa como una clase. Esto permite trabajar con objetos en lugar de instrucciones SQL escritas manualmente, y centraliza la definición del esquema en un único lugar: cualquier modificación —agregar una columna, renombrar un campo— se aplica una sola vez y se refleja en toda la aplicación.

\subsection{Esquemas de validación con Pydantic}

Cada operación de la API define un contrato: qué información acepta cuando el cliente envía datos, qué campos son obligatorios y qué estructura tiene la respuesta. Si una petición no cumple ese contrato —un precio negativo, un campo requerido ausente— se rechaza antes de llegar a la lógica de la aplicación, sin código de validación manual adicional.


\subsection{Autenticación y autorización}

El acceso al sistema se controla en dos pasos. El primero es la \textbf{autenticación}: al iniciar sesión, la API verifica la contraseña y, si es correcta, emite un \textbf{token de sesión} con vigencia de 30 minutos que contiene el identificador del usuario y sus permisos. Este token acompaña cada petición posterior; el servidor lo verifica sin consultar la base de datos, ya que la información necesaria está incluida y firmada dentro del propio token. Cuando expira, se renueva de forma automática sin que el usuario tenga que ingresar su contraseña de nuevo.

El segundo paso es la \textbf{autorización}: cada operación de la API declara qué permiso requiere para ejecutarse. Antes de procesar la petición, la API verifica que el token del usuario incluya ese permiso; si no lo tiene, la petición es rechazada. Esto conecta con la estructura de roles del OE1: un cajero puede registrar ventas pero no modificar usuarios ni configurar sucursales.

\subsection{Gestión de datos y formularios}

El estado de la aplicación se divide en dos responsabilidades. Las rutas de la aplicación están protegidas: si no hay sesión activa, el usuario es redirigido a la pantalla de inicio de sesión. Para operaciones que requieren permisos específicos se verifica además que el token incluya los permisos necesarios, de forma consistente con la estructura de roles del OE1.

\section{Desarrollo del nodo sensor con ESP32}

Para cumplir con el cuarto objetivo específico se diseñó e implementó un nodo electrónico que mide temperatura y humedad relativa en el área de almacenamiento y envía esas lecturas a la API del OE2. El nodo está integrado por un microcontrolador \textbf{ESP32-C5}, un sensor \textbf{AHT10} y tres pantallas \textbf{OLED SSD1306}, programado en C++ con el entorno Arduino.

\subsection{Selección del hardware}

Se eligió el \textbf{ESP32-C5} como microcontrolador por tres razones: integra conectividad WiFi de forma nativa sin necesitar un módulo adicional, soporta el protocolo I\textsuperscript{2}C necesario para comunicarse con el sensor y las pantallas, y es compatible con el entorno Arduino, que provee bibliotecas maduras para red y manejo de datos. Se descartó el Arduino Uno por carecer de WiFi integrado, y una Raspberry Pi habría resultado sobredimensionada para la tarea concreta de leer y enviar dos valores numéricos periódicamente, con un costo y consumo de energía mayores sin beneficio funcional correspondiente.

Para la medición ambiental se eligió el sensor \textbf{AHT10} sobre alternativas como el DHT11 o el DHT22 por dos razones: utiliza I\textsuperscript{2}C —el mismo bus compartido con las pantallas, sin necesitar conductores adicionales—, y ofrece una precisión de ±0.3\,°C en temperatura y ±2\% en humedad, adecuada para detectar variaciones relevantes en el almacenamiento de medicamentos donde los rangos recomendados son 15–25\,°C y 30–65\% de humedad relativa.

\subsection{Arquitectura del nodo}

El sensor AHT10 y las tres pantallas OLED se conectan al ESP32-C5 mediante el bus I\textsuperscript{2}C, que requiere únicamente dos conductores compartidos entre todos los dispositivos. La alimentación proviene de una fuente USB de 5\,V. La conexión con la API se realiza de forma inalámbrica a través de la red WiFi del establecimiento.

\subsection{Funcionamiento del firmware}

El firmware sigue el modelo de Arduino: una función de configuración inicial que se ejecuta una sola vez al encender el dispositivo —inicializa el bus I\textsuperscript{2}C, detecta el sensor y establece la conexión WiFi— y un ciclo operativo continuo que realiza las lecturas cada 30 segundos y las envía a la API. El ciclo utiliza temporización no bloqueante: en lugar de pausar el programa durante la espera, compara continuamente el tiempo transcurrido, lo que permite verificar el estado de la conexión WiFi en cada iteración.

Para la conexión WiFi se utilizó la biblioteca \textbf{WiFiManager}. En el primer arranque, el ESP32 crea una red temporal de nombre \texttt{Sensor-Farmacia}; el técnico se conecta desde un teléfono o computadora, selecciona la red del establecimiento e ingresa la contraseña. El dispositivo guarda esas credenciales y las usa automáticamente en arranques posteriores. Si se necesita cambiar la red —por ejemplo, al reubicar el dispositivo—, mantener presionado el botón de reinicio durante tres segundos borra las credenciales y reinicia el proceso de configuración.

Cada lectura se valida dentro de rangos físicamente plausibles (−10\,°C a 85\,°C para temperatura; 0\% a 100\% para humedad) antes de enviarse; si cae fuera de rango, se descarta sin enviar. Las lecturas válidas se envían al \textit{endpoint} \texttt{/api/v1/sensor-readings} mediante una petición POST con cifrado HTTPS. El intervalo de 30 segundos balancea la granularidad de los datos con la carga sobre la red: lecturas más frecuentes no aportarían información adicional relevante para el monitoreo ambiental de un almacén.

\subsection{Pantallas OLED}

Se incorporaron tres pantallas \textbf{OLED SSD1306} de 128×64 píxeles para que el personal pueda verificar el estado del nodo —temperatura, humedad y conexión WiFi— sin necesidad de abrir la aplicación web; cada pantalla muestra un único dato para que sea legible de un vistazo. Las tres comparten el mismo bus I\textsuperscript{2}C que el sensor AHT10 y operan cada una con una dirección de bus distinta para que el ESP32 pueda dirigirles instrucciones de forma individual. La actualización ocurre tras cada ciclo de lectura, reutilizando el mismo dato que se envía a la API sin operaciones adicionales.

\section{Integración física y lector de código de barras}

El sexto objetivo específico comprende dos componentes de integración: el diseño e impresión del gabinete físico que aloja el nodo sensor, y la incorporación de un lector de código de barras al flujo de registro de ventas en la aplicación web.

\subsection{Gabinete impreso en 3D}

Para alojar el ESP32, el sensor AHT10 y las tres pantallas OLED en un conjunto compacto y protegido, se diseñó y fabricó mediante \textbf{impresión 3D} un gabinete personalizado. Un gabinete a medida fue necesario porque la alineación horizontal de tres módulos OLED no corresponde a ninguna caja disponible comercialmente; una adaptación de una caja genérica habría requerido cortes imprecisos y un acabado poco profesional.

El gabinete se compone de dos piezas: una base que aloja la tarjeta del microcontrolador y el cableado interno, y una tapa frontal con tres ventanas rectangulares para exponer las pantallas OLED. Una abertura lateral permite la salida del cable de alimentación USB. Las dos piezas ensamblan a presión, sin tornillos.

Las decisiones de diseño respondieron a restricciones concretas. El ensamble a presión se eligió porque los componentes internos pueden requerir ajuste durante la vida útil del dispositivo: con tornillos, el acceso repetido desgasta las roscas en plástico impreso; con el encaje a presión, abrir y cerrar el gabinete no deteriora el mecanismo. El ancho mínimo lo determinaron los tres módulos OLED dispuestos en fila horizontal con sus conectores orientados hacia la base. La ventilación del sensor AHT10 se resolvió con una ranura en la base, suficientemente pequeña para evitar la entrada de polvo pero que permite la circulación del aire del área de almacenamiento.

\subsection{Integración del lector de código de barras}

Para agilizar el registro de ventas se incorporó un lector de código de barras USB que el sistema operativo reconoce como un teclado estándar; desde la perspectiva del navegador, el código escaneado se ingresa como si hubiera sido tecleado manualmente, sin requerir ningún controlador especial.

El flujo estándar de creación de venta no estaba diseñado para el ritmo de escaneo continuo: el operador habría tenido que hacer clic en el campo correcto y esperar el resultado por cada artículo. Para que la herramienta resultara práctica en un punto de venta, se diseñó un \textbf{modo lector} en el módulo de ventas, activable desde cualquier parte del formulario con la tecla \textbf{F2}, que redirige el foco al campo de lectura sin que el usuario tenga que desplazarse.

Al escanear un código, la aplicación lo compara contra el catálogo de productos cargado en memoria al abrir el formulario. Si hay coincidencia, el producto se agrega a la lista de artículos de la venta; si no la hay, el campo muestra una señal de error. La búsqueda se resuelve localmente sin generar una petición al servidor por cada código, lo que evita la latencia de red en un flujo donde múltiples artículos se escanean en secuencia rápida. El modo se desactiva automáticamente cuando el usuario hace clic fuera del área de escaneo.
